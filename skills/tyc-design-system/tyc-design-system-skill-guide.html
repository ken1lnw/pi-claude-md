1ภาพรวม

ปัญหาที่ skill นี้แก้: ทุกครั้งที่สร้างหรือแก้ระบบใหม่ Claude มักจะ "เดา" สี ขนาดมุมโค้ง หรือรูปแบบปุ่มขึ้นมาเอง ทำให้แต่ละระบบภายในบริษัทหน้าตาไม่ตรงกัน

Skill tyc-design-system แก้ปัญหานี้ด้วยการบังคับให้ Claude อ่านค่าจาก token/ไฟล์อ้างอิงจริงก่อนเขียนโค้ดทุกครั้ง แทนที่จะจำหรือเดาค่าสีจากบทสนทนาเก่า และมีกติกาตายตัว (ปุ่มต้องมน, หัวตารางต้องฟ้า, ชมพูใช้แค่จุดเน้น ฯลฯ) ที่ Claude ต้องทำตาม เว้นแต่จะถามผู้ใช้ก่อน

Skill นี้เป็น personal skill ผูกกับเครื่องนี้เครื่องเดียว (ดูหัวข้อ "การติดตั้ง" ว่าจะขยายไปทั้งทีมอย่างไร)

2การติดตั้ง

สำหรับเครื่องนี้ (tit1622) ติดตั้งเรียบร้อยแล้ว ไม่ต้องทำอะไรเพิ่ม — หัวข้อนี้มีไว้สำหรับตอนย้ายเครื่อง หรือแจกจ่ายให้เพื่อนร่วมทีม

2.1 ติดตั้งแล้วบนเครื่องนี้

C:\Users\tit1622\.claude\skills\
  └─ tyc-design-system\
     ├─ SKILL.md
     ├─ tyc-design-system-skill-guide.html ← ไฟล์นี้

คู่มือฉบับนี้อยู่ในโฟลเดอร์ skill โดยตรงแล้ว — เวลาคัดลอกหรือแจกจ่ายทั้งโฟลเดอร์ tyc-design-system\ (ดูหัวข้อ 2.2 และ 2.3) จะได้คู่มือนี้ติดไปด้วยเสมอ ไม่ต้องส่งแยก

2.2 ติดตั้งให้เพื่อนร่วมทีม (คนละเครื่อง)

Personal skill ไม่ sync ข้ามเครื่องอัตโนมัติ ต้องคัดลอกโฟลเดอร์ทั้งหมดไปวางในตำแหน่งเดียวกันบนเครื่องของเพื่อนร่วมทีม:

PowerShell# บนเครื่องปลายทาง — วางโฟลเดอร์ tyc-design-system ทั้งก้อนไว้ที่นี่
New-Item -ItemType Directory -Force "$env:USERPROFILE\.claude\skills" | Out-Null
Copy-Item -Recurse ".\tyc-design-system" "$env:USERPROFILE\.claude\skills\tyc-design-system"

ส่งโฟลเดอร์ tyc-design-system ให้ผ่านทาง shared drive, USB หรือ zip แล้วส่งใน Teams/Slack ก็ได้ — ไม่มี dependency ภายนอก เป็นไฟล์ข้อความล้วน

2.3 ติดตั้งแบบผูกกับ repo (แนะนำถ้าต้องการบังคับทั้งทีมอัตโนมัติ)

วิธีข้างบนต้องคัดลอกเองทีละเครื่อง ถ้าต้องการให้ ทุกคนที่ clone repo ได้ skill ไปด้วยทันทีโดยไม่ต้องติดตั้งเอง ให้คัดลอกโฟลเดอร์เดียวกันไปไว้ใน path ของ repo แทน แล้ว commit เข้า git:

ตัวอย่างสำหรับ repo wmsC:\Users\<user>\bc_project\wms\.claude\skills\tyc-design-system\
  ├─ SKILL.md
  └─ reference\ ...

ข้อดี: อัปเดตครั้งเดียวใน repo ทุกคนได้ของใหม่ผ่าน git pull ปกติ ไม่ต้องแจกไฟล์เอง — เหมาะกับเป้าหมาย "ทั้งองค์กรใช้ธีมเดียวกัน" มากกว่าแบบ personal

ถ้าติดตั้งแบบผูกกับ repo ต้องคอย sync กับต้นฉบับที่ ~/.claude/skills/tyc-design-system/ เองเวลามีการแก้ theme (ดูโหมด extend ด้านล่าง) ไม่งั้นแต่ละ repo จะเห็นเวอร์ชันธีมไม่ตรงกัน

3ตรวจสอบว่าใช้งานได้

เปิด Claude Code session ใหม่ในโปรเจกต์ไหนก็ได้ แล้วลองถามตรงๆ ว่า:

"มี skill อะไรให้ใช้บ้าง"

ถ้าเห็น tyc-design-system ในรายการที่ตอบกลับมา แปลว่าติดตั้งถูกต้อง — หรือจะข้ามขั้นตอนนี้แล้วลองสั่งงานตรงๆ ตามหัวข้อถัดไปเลยก็ได้ เพราะ skill นี้ถูกออกแบบให้เรียกใช้อัตโนมัติเมื่อคำขอเข้าเงื่อนไข ไม่ต้องเปิดเองก่อนเสมอไป

4วิธีเรียกใช้

ไม่มีคำสั่งพิเศษที่ต้องจำ — พิมพ์สิ่งที่ต้องการเป็นภาษาปกติ Claude จะจับคู่กับ skill นี้เองถ้าเนื้อหาเข้าเงื่อนไข (พูดถึง "TYC theme", "design system", หรือทำ UI ให้ระบบภายในองค์กร)

"ปรับหน้านี้ให้ตรงกับ TYC design system หน่อย"

ถ้าอยากเจาะจงให้ชัวร์ว่าเรียก skill นี้แน่ๆ ให้พิมพ์ชื่อ skill ตรงๆ ได้เหมือนกัน:

/tyc-design-system full

ด้านล่างนี้คือ 3 โหมดที่ skill รองรับ — พิมพ์เป็นประโยคปกติแล้วระบุขอบเขตให้ชัดก็พอ ไม่จำเป็นต้องพิมพ์ชื่อโหมดตรงๆ

full

เปลี่ยน theme ทั้งหมด

รีสกินทุกหมวด (สี ปุ่ม ตาราง การ์ด ฟอร์ม navigation dark/light) ทั้งระบบในครั้งเดียว เหมาะกับระบบใหม่ หรือระบบเก่าที่ยังไม่เคยตรง theme เลย

"เปลี่ยน theme ทั้งระบบนี้ให้เป็น TYC design system"
partial

เปลี่ยน theme บางส่วน

ระบุเฉพาะจุดที่ต้องการ เช่น สีหรือปุ่ม โดยไม่แตะส่วนอื่น ถ้าบอกแค่ "บางส่วน" โดยไม่ระบุ Claude จะถามกลับให้เลือกจากเมนู 11 หมวด (ดูด้านล่าง)

"เปลี่ยนแค่ปุ่มกับตารางให้ตรง TYC theme"
extend

เพิ่ม pattern ใหม่เข้า theme

ไม่ใช่การแก้ระบบปลายทาง แต่เป็นการเพิ่ม component/token ใหม่เข้าไปใน theme-ui-guideline.html ต้นฉบับ แล้ว sync กลับเข้า skill ให้อัตโนมัติ

"เพิ่ม component นี้เข้าไปใน TYC design system ด้วย"

เมนู 11 หมวดสำหรับโหมด partial

เมื่อ Claude ถามว่าจะเปลี่ยนส่วนไหน จะเลือกได้จากรายการนี้เท่านั้น (ตรงกับ section ใน theme-ui-guideline.html แบบ 1:1):

  • Colors & Tokens
  • Typography
  • Radius & Spacing
  • Buttons
  • Tables
  • Cards & Layout
  • Forms & Inputs
  • Navigation
  • Overlays
  • Charts
  • Dark/Light mode

5โครงสร้างไฟล์อ้างอิง

Claude จะอ่านไฟล์เหล่านี้ก่อนเขียนโค้ดเสมอ แทนการจำค่าจากบทสนทนาเก่า

ไฟล์ใช้ทำอะไร
tokens-and-rules.mdCheat sheet ค่าสี/radius/spacing/dark-mode แบบย่อ — จุดแรกที่ถูกอ่านเสมอ
component-conventions.mdโครงสร้าง badge สถานะ (พร้อมตารางแปลความหมาย), SVG ไอคอนของปุ่ม action ทุกแบบ (ดู/แก้ไข/ลบ/เพิ่ม/รีเฟรช ฯลฯ), และโครงสร้าง navbar/header ที่ตายตัว (ลำดับปุ่ม, สีพื้นหลัง) — จุดที่ token สีอย่างเดียวไม่พอ ต้องเป๊ะทั้งรูปทรง/โครงสร้างด้วย
theme-ui-guideline.htmlComponent library ฉบับเต็ม เปิดดูในเบราว์เซอร์ได้ แต่ละ component มีปุ่ม "ดูโค้ด" คัดลอกไปวางได้ทันที
tyc-design-tokens.cssToken แบบ Tailwind v4 (CSS-first) — schema ตรงกับ frontend-mes / rcs-rmr-pm / wms
tyc-design-tokens.jsonToken แบบ W3C JSON — ใช้กับโปรเจกต์ที่ไม่ใช่ Tailwind
tailwind.config.jsสำหรับโปรเจกต์ Tailwind v3 (รุ่นเก่าที่ยังไม่ย้ายเป็น CSS-first)
ไฟล์เหล่านี้เป็น "สำเนา" — ต้นฉบับจริงอยู่ที่ C:\Users\tit1622\Desktop\theme-ui-guideline.html ถ้าแก้ต้นฉบับแล้วลืม sync กลับเข้า reference\ โหมด full/partial ครั้งต่อไปจะยังใช้ข้อมูลเก่าอยู่ — ใช้โหมด extend เพื่อให้ sync ให้อัตโนมัติ

6กติกาที่ห้ามยืดหยุ่น

Claude ต้องทำตามข้อเหล่านี้เสมอ เว้นแต่จะถามและได้รับอนุญาตจากผู้ใช้ก่อน

  • ปุ่มทุกแบบต้องเป็นทรงมน (pill) ยกเว้นปุ่มไอคอนเดี่ยวที่ยังเป็นเหลี่ยมมน (radius-md)
  • หัวตารางต้องพื้นฟ้าแบรนด์ ตัวอักษรขาวเสมอ
  • สีชมพู (accent) ใช้ได้แค่จุดตกแต่ง ไม่เกิน ~25% ของหน้าจอ ห้ามใช้กับปุ่มหลักหรือข้อความยาว
  • ห้าม hardcode เลขสี (hex) ใหม่ในโค้ด — ทุกสีต้องมาจาก token เท่านั้น เพื่อให้ dark mode ทำงานถูกต้องเสมอ
  • ระบบใหม่ทุกระบบต้องมี dark/light mode ตั้งแต่เริ่มสร้าง ไม่ใช่ค่อยเพิ่มทีหลัง
  • ถ้าระบบเดิมมี convention ที่ขัดกับ theme นี้ Claude ต้องหยุดถามก่อนเสมอ ห้ามเขียนทับเงียบๆ

7คำถามที่พบบ่อย

?สั่งงานแล้ว Claude ไม่ได้ใช้ theme นี้ ทำยังไง

พิมพ์ชื่อ skill ให้ชัดเจนขึ้น เช่น "ใช้ tyc-design-system ปรับ..." หรือพิมพ์ /tyc-design-system full ตรงๆ เพื่อบังคับเรียกโดยไม่ต้องพึ่งการจับคู่อัตโนมัติ

?แก้ไฟล์ theme-ui-guideline.html บน Desktop เพิ่มเติมแล้ว ต้องทำอะไรต่อ

ถ้าสั่งผ่านโหมด extend Claude จะ sync ไฟล์กลับเข้า reference/ ให้เองเป็นขั้นตอนสุดท้าย แต่ถ้าแก้ไฟล์เองนอกบทสนทนา ต้องคัดลอกไฟล์ที่เปลี่ยนไปทับใน reference/ ด้วยตัวเอง ไม่งั้น skill จะยังอ้างอิงเวอร์ชันเก่า

?ทีมอื่นที่ไม่ได้อยู่เครื่องนี้ใช้ skill นี้ได้ไหม

ไม่โดยอัตโนมัติ — เป็น personal skill ผูกกับเครื่อง tit1622 เท่านั้น ต้องคัดลอกโฟลเดอร์ไปติดตั้งที่เครื่องอื่น (ดูหัวข้อ "การติดตั้ง") หรือย้ายไปผูกกับ repo แทนถ้าต้องการให้ทุกคนที่ clone repo ได้อัตโนมัติ

?ระบบเดิมมีสี/ปุ่มไม่เหมือน TYC theme อยู่แล้ว Claude จะเขียนทับเลยไหม

ไม่ — กติกาบังคับให้ Claude หยุดถามก่อนเสมอเมื่อเจอ convention เดิมที่ขัดกับ theme (ดูหัวข้อ 6) เพื่อป้องกันการรีสกินทับการตัดสินใจเดิมของทีมโดยไม่ตั้งใจ

?อยากเพิ่ม component ใหม่เข้า design system ต้องทำยังไง

ใช้โหมด extend — บอก Claude ว่าต้องการเพิ่มอะไรเข้า TYC design system ระบบจะเพิ่มเข้าไฟล์ต้นฉบับตามรูปแบบเดิม (การ์ด + โค้ดตัวอย่าง) ตรวจสอบสีถ้ามีการเพิ่มสีใหม่ แล้ว sync กลับเข้า skill ให้เอง

?สั่งปรับ theme แล้ว badge สถานะ ปุ่ม action (ดู/แก้ไข/ลบ) หรือ navbar ยังไม่เหมือนต้นฉบับ ทำไม

เพราะ 3 จุดนี้ต้องเป๊ะทั้ง "รูปทรง/โครงสร้าง" ไม่ใช่แค่สี — badge ต้องมีจุดนำหน้าอัตโนมัติจาก .badge::before ไม่ใช่ไอคอนที่แทรกเอง, ปุ่ม action ต้องใช้ SVG เดียวกับต้นฉบับเป๊ะๆ (path data เดียวกัน), และ navbar ต้องพื้นฟ้าแบรนด์ + ลำดับปุ่มตรงกัน (สลับธีม → รีเฟรช → ภาษา → โปรไฟล์) navbar เป็นจุดที่มักถูกข้ามเพราะอยู่เหนือจอตอนทำงานส่วนอื่น จุดนี้แก้แล้วโดยเพิ่มไฟล์ component-conventions.md ที่มีโค้ดตัวอย่างครบทั้ง 3 เรื่องไว้ให้คัดลอกตรงๆ — ถ้ายังไม่ตรงหลังจากนี้ ให้บอก Claude อ้างอิงไฟล์นี้ตรงๆ
